# REALTIME.md - Arsitektur Real-Time & Sinkronisasi Klien
⚡ REALTIME.md - Arsitektur Real-Time & Sinkronisasi Klien Dokumen ini mendefinisikan standar komunikasi WebSocket, struktur payload, strategi sinkronisasi cache, dan penanganan kegagalan untuk fitur real-time di aplikasi Kanban menggunakan Supabase Realtime dan TanStack Query.

## Arsitektur Utama & Source of Truth
Sistem tidak menggunakan mode Postgres Changes bawaan untuk menghindari keterikatan skema basis data mentah dengan klien. Source of Truth: Database PostgreSQL selalu menjadi source of truth (sumber kebenaran utama). Event real-time hanya berfungsi sebagai mekanisme sinkronisasi cache UI dan tidak boleh dianggap sebagai sumber data utama. Jika terjadi konflik antara cache lokal, event real-time, dan hasil kueri database, maka hasil kueri database harus selalu diprioritaskan. Event Ordering: Sistem tidak menjamin urutan kedatangan event real-time antar klien. Klien tidak boleh mengasumsikan bahwa dua event yang dikirim server akan selalu diterima dalam urutan yang sama. Jika terjadi ketidaksesuaian state akibat urutan event atau event yang terlewat, mekanisme Full Re-sync dan Database Source of Truth menjadi jalur pemulihan resmi. Realtime di atas jaringan publik tidak pernah dapat menjamin ordering sempurna. Mode Siaran (Broadcast): Klien berlangganan ke saluran khusus per ruang kerja dengan pola workspace:{workspaceId}. Penyiaran event dieksekusi secara eksklusif oleh Server Actions setelah transaksi basis data sukses melakukan commit. Otorisasi Saluran: Menggunakan RLS (Row Level Security) minimal pada PostgreSQL yang diaktifkan murni untuk melindungi publikasi Realtime. Supabase akan otomatis menolak klien yang mencoba berlangganan jika user_id mereka tidak terdaftar di tabel workspace_members.

## Batasan Scope & Non-Goals MVP
Untuk mencegah pelebaran lingkup (scope creep) dan lonjakan konsumsi arsitektur pada fase MVP, sistem ini memiliki batasan yang tegas. Batasan Presence MVP: Supabase Presence HANYA digunakan untuk: Status Online pengguna. Status Offline pengguna. Menampilkan anggota workspace yang aktif (Active workspace members / Avatar Stack). Non-Goals (MVP): Sistem realtime MVP tidak mencakup: Event replay Event history Message persistence Exactly-once delivery Distributed locking Conflict-free replicated data types (CRDT) Collaborative document editing Live cursor, live dragging, live typing indicator Kebutuhan tersebut ditunda hingga terdapat kebutuhan produk yang nyata di V2. Ini membantu menjaga arsitektur tetap sederhana.

## Standar Event Envelope, Versioning & Metadata
Untuk menjamin konsistensi handler di sisi klien dan kemudahan logging, setiap siaran real-time wajib mematuhi standar kontrak tunggal berikut: type RealtimeEvent = { version: 1; eventId: string; mutationId?: string;
workspaceId: string; actorId: string;
type: EventType;
// Tipe payload dilonggarkan (loose) di kontrak arsitektur ini. // Definisi union ketat akan didokumentasikan di kode (realtime.types.ts). payload: Record<string, unknown>;
timestamp: string; // ISO 8601 UTC (contoh: "2026-06-04T12:34:56.789Z") } Payload Guidelines: Payload real-time harus berisi data minimum yang diperlukan untuk memperbarui cache klien. Payload tidak boleh digunakan untuk mengirim ulang keseluruhan board, daftar task penuh, atau struktur workspace lengkap. Data besar harus diambil melalui HTTP query normal. Ini menjaga biaya dan latensi real-time tetap rendah. Workspace Isolation (Pertahanan Berlapis): Klien wajib mengabaikan event yang memiliki workspaceId berbeda dengan workspace yang sedang aktif. Walaupun channel sudah diisolasi per workspace, validasi tambahan di sisi klien memberikan lapisan pertahanan ekstra terhadap kesalahan konfigurasi atau bug integrasi. if (event.workspaceId !== currentWorkspaceId) { return; } Aturan Versioning: Field version digunakan untuk menjaga kompatibilitas kontrak event di masa depan. Perubahan payload yang bersifat breaking change wajib menaikkan versi event envelope. Klien yang menerima versi yang tidak dikenali wajib mengabaikan event tersebut dan melakukan logging untuk observability: Catatan Metadata: eventId wajib unik global. Digunakan untuk observability, tracing log, dan deduplikasi. mutationId berasal dari klien. Digunakan murni untuk mencegah echo effect. Jangan mencampur fungsi keduanya.

## Strategi Mencegah Echo Effect (Duplicate Events)
Karena klien menggunakan Optimistic Updates instan, event broadcast dari peladen yang memantul kembali ke klien pemicu dapat menyebabkan UI flickering. Ini dicegah menggunakan Client Mutation ID. Implementasi Frontend: Klien menghasilkan mutationId (UUID) pada setiap pemanggilan Server Action dan menyimpannya ke dalam Set<string> (atau Map<string, timestamp>) lokal bernama pendingMutationIds. Listener Klien (Pseudocode): Pembersihan (TTL): ID di dalam struktur data lokal akan dibersihkan secara otomatis setelah 30–60 detik.

## Strategi Rekoneksi (Lost Events & Recovery)
Mencoba menyusun ulang riwayat event tunggal saat jaringan terputus sangatlah berisiko. Strategi yang digunakan adalah "Wipe and Fetch" (Full Re-sync). Perilaku Klien: Ketika saluran WebSocket berhasil tersambung kembali setelah kondisi terputus (disconnected), klien wajib memaksa TanStack Query membuang cache dan menarik status board utuh terbaru dengan mengeksekusi queryClient.invalidateQueries(). Optimasi: Implementasi wajib menggunakan mekanisme debounce atau pengecekan timestamp lokal (misal: jeda minimal 5 detik antar penarikan) agar koneksi yang tidak stabil (naik-turun) tidak memicu spam request ke peladen.

## Passive Cache Recovery (Dual-Write Safety Net)
Terdapat kemungkinan kecil di mana transaksi database berhasil namun pengiriman broadcast gagal. TanStack Query wajib dikonfigurasi sebagai fallback consistency mechanism. Realtime adalah jalur sinkronisasi utama, sementara HTTP refetch adalah jalur pemulihan: // Konfigurasi wajib TanStack Query { staleTime: 5 * 60 * 1000, // 5 Menit refetchOnWindowFocus: true, refetchOnReconnect: true, }

## Siklus Hidup Ekstrem (WORKSPACE_DELETED Flow)
Penghapusan ruang kerja adalah operasi destruktif yang menghancurkan keseluruhan konteks saluran. Basis data wajib selalu menjadi source of truth. Alur Wajib Penghapusan Workspace: Server melakukan Soft Lock pada Workspace (mencegah mutasi baru). Database Transaction: Melakukan Hard Delete pada PostgreSQL. Broadcast Event: Menyiarkan event WORKSPACE_DELETED ke seluruh klien yang masih terhubung. Disconnect: Menutup koneksi klien (opsional/didelegasikan ke sistem). Catatan Ketergantungan Broadcast: Jika broadcast gagal setelah penghapusan berhasil dilakukan di basis data, mekanisme Passive Cache Recovery (seksi 7) tetap akan mengembalikan klien ke state yang benar melalui refetch berikutnya. Dengan demikian tidak ada ketergantungan absolut pada keberhasilan broadcast. Penanganan Klien: Saat klien menerima WORKSPACE_DELETED, klien wajib: membersihkan cache (queryClient.clear()), melakukan redirect ke /dashboard, dan menutup koneksi secara anggun (channel.unsubscribe()).
